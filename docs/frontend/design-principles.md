# Frontend Design Principles

この文書は，`geshi` の frontend で見栄えや操作ラベルを調整するときの，
日常的な判断基準をまとめたものである．

ADR のような大きな設計判断ではなく，UI の細かな調整で迷ったときに参照する．

## 目的

- 画面ごとの調整が場当たり的になることを避ける
- 実装都合の語彙を UI にそのまま出さない
- 操作の意味が，コードを知らなくても分かる状態を保つ
- 見た目の一貫性より，誤解されにくさを優先する
- 基本的に自分用の UI であることを踏まえ，頻用操作は素早く触れられることを優先する

## ラベル

- UI の操作ラベルは，内部実装の名前ではなく，利用者が受け取る結果を優先して付ける
- `observe` のような内部 API 名や job 名は，そのまま UI に出さない
- 短くできる場合でも，意味が曖昧になるなら短縮しない
- 並ぶ操作の語形は，機械的に揃えすぎない
  - 見た目の対称性より，ひと目で見分けられることを優先する

例:

- source を実際に見に行く操作は `Observe` ではなく `Crawl`
- 保存済み一覧を再取得するだけの操作は `Refresh` より `Reload`
- source を追加する操作は，pane header に直接置く `+` でもよい

## アイコン

- 頻用操作は，icon-only で直接置いてよい
- その icon が自分にとって十分に自明であり，操作対象との距離も近い場合は，text を省略してよい
- 低頻度操作や意味の判別が必要な操作は，text を付ける
- icon を付けても意味が明確にならない場合は，使わない
- 装飾目的だけで icon を足さない
- 同じ menu や同じ操作群では，情報量を揃える
- 一揃いの menu の中で，ひとつだけ icon 付き，あるいはひとつだけ icon 無しという状態は避ける
- 一部だけ意味の弱い icon を付けるくらいなら，全体をテキストだけに寄せる
- icon-first ではなく label-first で考える

## メニュー

- メニューには，その文脈で使う操作だけを置く
- 頻用操作は menu の外に直接置き，低頻度操作は menu にまとめる
- 画面の再読込みと，実際の処理開始は分けて表現する
- 危険操作や重い操作でなくても，結果の違う操作は近い名前に寄せすぎない
- 項目数が少ない menu では，簡潔さより識別しやすさを優先する

## 配置

- 操作は，できるだけ対象の近くに置く
- global な操作は page header，local な操作は pane や section header に置く
- 特定 pane 専用の操作を page 全体のヘッダに置かない
- 「何に効く操作か」が位置だけでも伝わる配置を優先する

例:

- source 追加は app 全体の header ではなく，Sources pane header に置く

## 文言の基準

- まず英語 UI として自然であることを優先する
- 将来的な i18n を見越し，意味の境界が曖昧な語を避ける
- 翻訳しやすいように，実装依存の略語や比喩を多用しない
- loading 中の文言も，開始された処理が分かるようにする

例:

- `Crawl`
- `Crawling...`
- `Reload`

## 一貫性の考え方

- すべてを同じ見た目にすることを，一貫性とはみなさない
- 操作の種類や重さが違うなら，ラベルや配置が多少非対称でもよい
- 一貫性は「同じ意味のものが同じように見えること」であり，「違う意味のものまで同じ見た目に寄せること」ではない

## 運用

- frontend の見栄えや操作文言で迷ったら，まずこの文書に照らして判断する
- 個別画面の事情で例外を入れる場合は，その理由を PR や commit に短く残す
- 原則自体を更新した方がよい変更が増えたら，この文書へ追記する

# Entry List Pagination

この開発項目では，geshi の entry 一覧取得が全件一括取得ではなくページ単位で取得できることを受け入れ条件とする．
entry は永続化モデル上 `content` として保持されているが，browse UI と API では利用者向けに entry 一覧として扱う．
新しい entry が継続的に増える前提でも，一覧の表示順を保ちながら，段階的に続きを取得できることを今回の完了条件とする．

## 受け入れ条件

- backend に，entry 一覧取得 API のページング仕様が追加される
- entry 一覧取得 API は，先頭ページ取得と続きページ取得の両方を扱える
- page size は API で制御でき，未指定時の default と過大要求時の上限が定義される
- entry 一覧の並び順は現行の新しい順を維持する
- 続きページは，直前ページの末尾を基準にして重複なく取得できる
- 新しい entry が先頭側に追加されても，既に取得済みの続きを辿る経路で entry の取りこぼしや重複を起こしにくい方式が採られる
- page 境界をまたぐときも，同一 sort key を持つ entry が順序不定にならない
- 一覧 API の response には，entry 配列に加えて続きを取得するための pagination metadata が含まれる
- 続きがない場合は，それを判定できる response を返せる
- frontend は，entry 一覧の初回表示で先頭ページだけを取得する
- frontend は，利用者操作または scroll 契機で続きを取得し，既存一覧の末尾へ追加できる
- frontend は，追加読込中，続きを読み切った状態，追加読込失敗時の状態を扱える
- entry 一覧から個別 entry detail を開く既存導線は，ページング導入後も維持される
- backend / frontend の主要な正常系と異常系について，自動確認が追加されている

## 確認方法

- entry が page size を超える件数ある状態で，先頭ページが指定件数以内で返ることを確認する
- 先頭ページ response の pagination metadata を使って続きを取得できることを確認する
- 2 ページ以上を連続取得しても，同じ entry が重複せず，順序が新しい順のまま連結できることを確認する
- 同じ `publishedAt` を持つ entry が複数ある場合でも，ページ境界をまたいで順序が安定することを確認する
- 先頭ページ取得後に新しい entry が追加されても，既存 cursor で次ページを取得したときに既取得 entry の重複が増えないことを確認する
- 続きがない場合に，frontend が追加取得を停止できることを確認する
- 不正な page size や cursor を与えたときの API error が定義どおりであることを確認する
- frontend で初回表示，追加取得，末尾到達，追加取得失敗の主要状態が自動テストで確認されることを確認する

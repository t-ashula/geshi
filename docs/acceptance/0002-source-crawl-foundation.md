# Source Crawl Foundation

この開発項目では，登録済み `podcast rss source` を起点にクロールを実行し，しばらく待つと収集結果の `content` 一覧を確認できる最小経路を，job queue と plugin 境界を含めて利用可能な状態まで実装することを受け入れ条件とする．

## 受け入れ条件

- `source` からクロールを開始するために必要な前提判断が ADR として整理され，レビューを経て `決定` になっている
- podcast RSS source の登録後に，初回クロール job を backend 経由で投入できる
- job queue を介して，投入されたクロール job が非同期に実行される
- クロール処理の実行結果として，少なくとも成功 / 失敗と再試行可能性を追跡できる
- クロール本体は plugin 境界を持ち，`source` に応じて実行ロジックを差し替えられる
- 既存の podcast RSS source は，plugin 経由でクロールできる
- 正常系では，登録済み source に対するクロール完了後，少なくとも収集した `content` と実行記録が一貫して残る
- frontend または backend 経由で，しばらく待った後に収集済み `content` 一覧を確認できる
- 異常系では，plugin の失敗，取得失敗，保存失敗の少なくともいずれかに対して，job の状態と失敗理由を利用者または運用者が追跡できる
- クロール基盤の主要な正常系と異常系が，自動テストまたは再現可能な確認手順で検証されている
- 継続クロールを積み続ける scheduler job の詳細設計は，この開発項目の受け入れ条件には含めない

## 確認方法

- `docs/decisions/` に，クロール基盤に必要な ADR が揃い，対象 ADR のステータスが `決定` になっていることを確認する
- podcast RSS source 登録後に初回クロール job を投入し，queue を介して worker が非同期実行することを確認する
- podcast RSS source を対象にクロールを実行し，plugin 経由で取得処理が行われることを確認する
- 正常完了後，収集した `content` と job の実行記録が対応づいて残ることを確認する
- しばらく待った後に，frontend または backend の一覧取得経路から収集済み `content` 一覧を確認できることを確認する
- 取得失敗または保存失敗を発生させ，job の状態遷移と失敗理由を追跡できることを確認する
- 主要な正常系と異常系を扱う自動テスト，または同等の確認手順が追加されていることを確認する

# 0030 models

## 位置づけ

この文書は，データモデル再設計について `ADR-0030` に至るまでの検討メモである．

ただし，`0030` で扱う範囲は途中で絞り込み，最終的にはモデルの中身を決めるのではなく，モデル設計と文書化の層分離原則を導入するところまでを対象にする．

## 仕切り直し後の結論

`0030` では，個別の model 名や関係を決定しない．

今回決めるのは次の点だけである．

- 概念的なモデルとその関係は ADR で扱う
- 必要な情報や API レベルのインタフェースは [`docs/models.md`](../models.md) で扱う
- 永続化の物理構造はコードで示す

つまり，`0030` はデータモデルの具体案を決める ADR ではなく，今後どの層に何を書くかを決める ADR として閉じる．

## ここまでの検討で見えたこと

途中の検討では，少なくとも次の論点が独立して存在することが見えた．

- backend / frontend から見た機能要件の整理
- 公開単位，配信元，成果物，非同期処理の責務分離
- 収集処理の plugin 化を入れたときの境界
- 検索用の集約 view と本体 model の境界

これらは重要だが，`0030` の中でまとめて決めるには範囲が広すぎる．

## `0030` で決めないこと

次の点は，この design log では検討したが，`0030` の決定対象からは外す．

- 具体的な model 一覧
- model 間の関係
- model 名
- plugin 境界のために追加すべき概念の有無
- `docs/models.md` に書くべき個別インタフェースの詳細

## 次の進め方

`0030` を閉じた後は，必要に応じて別の design log または ADR で，個別論点を小さく分けて進める．

候補としては，少なくとも次がある．

- 公開単位と上位まとまりの整理
- 成果物と派生データの整理
- 収集処理 plugin 化に伴う境界整理
- 検索用集約 view の整理

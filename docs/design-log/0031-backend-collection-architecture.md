# 0031 backend collection architecture

## 位置づけ

この文書は，`ADR-0031` の検討メモ置き場である．

## 情報収集として想定していること

### 全般

backend での情報収集の基本モデルとして，「エントリー」を仮置きする．これは，podcast における 1 episode とほぼ同義である．

backend での情報収集は，エントリーの一覧を継続的に観察して，未取得のエントリーについて，個別エントリーとして取得（ダウンロード・録画）し，geshi 内で利用可能にすることである．

エントリーの一覧の観察方法としては，RSS の定期的な取得や，html の定期的なスクレイピング，API の定期的な実行などがありえる．

個別のエントリーの取得方法についても，単純なダウンロードや認証ありのアクセス．予定時間での hls などのストリームの取得などがありえる．
これらのエントリーの観察方法と取得方法をひとまとまりにしたものをチャンネルとする．

また，観察方法と取得方法とを geshi 本体の実装とは切り離し，プラグイン構成を取る．ただし，いくつかの単純な podcast などは標準装備とする．

### job との関係

情報収集の workflow は，既存の [`job`](../job.md) model の上で動かす前提で考える．

少なくとも，次の種類の `job` が必要になる．

1. 投入ジョブ
2. 観察ジョブ
3. 取得ジョブ

現時点では，それぞれの責務を次のように捉えるのがよさそうである．

- 投入ジョブ
  - 定期実行される
  - 実行対象となる `channel` を拾い，観察ジョブを登録する
- 観察ジョブ
  - 対象は `channel`
  - `channel` にぶら下がる `collector` を使って，個別対象を見つける
  - 新規の `entry` を予約状態で backend に書き戻す
  - 取得予定時刻を判定し，即時実行の取得ジョブまたは `scheduled` な取得ジョブを登録する
- 取得ジョブ
  - 対象は `entry`
  - `entry` が属する `channel` と，そこにぶら下がる `collector` を使って，実データ取得を行う
  - 成功時は `asset` を保存し，失敗時は `entry` を失敗確定に進める

`job.md` との対応で重要なのは，これらはすべて Geshi 側 `job` として登録され，BullMQ 側の runtime job とは分けて扱う点である．

つまり，

- 収集系 workflow でも，まず Geshi 側 `job` を永続化する
- 実行可否判断，queue 投入，実行状態反映，終端状態反映は [`job`](../job.md) の流れに従う
- 収集系独自に必要なのは，`kind` と `payload` の切り方である

### `job.target` についての見直し

現時点の収集系 workflow では，job を単純に「ある resource に対する job」として結び付ける前提は置かない方がよい．

理由:

- 投入ジョブは，特定の `channel` や `entry` 1 件に素直にぶら下がらない
- 観察ジョブも，実際には `channel` 本体だけではなく，そこにぶら下がる `collector` 設定や実行時条件を前提に動く
- 取得ジョブも，`entry` だけを見れば足りるとは限らず，時刻条件や plugin 設定，実行時の判断が絡む
- したがって，「どの resource に対する job か」という単純な紐づけを model の中心に置くと，かえって実態を表しにくい

このため，収集系 workflow では少なくとも，

- `kind`
- `payload`
- `runAfter`

を中心に job を定義し，resource への単純参照を前提にしない方が自然である．

この論点は，既存の `job` model 側の `target` 記述と衝突するので，後続で `job` 側文書と ADR の見直しが必要になる．

### 取得ジョブを作るタイミング

現時点では，取得ジョブは観察ジョブが作る前提にするのがよい．

- 即時取得できる対象なら，観察ジョブが即時実行の取得ジョブを登録する
- 予定時刻まで待つ必要がある対象なら，観察ジョブが `runAfter` を持つ `scheduled` な取得ジョブを登録する

この整理にすると，

- 個別対象の発見と次段の起動判断が同じ文脈で完結する
- 録画予約のようなケースも，別の取得投入処理を増やさずに扱える
- 実際の起動は既存の `scheduler job` が担うので，worker 数調整の整理とも矛盾しない

今後の論点は，取得ジョブの重複登録防止と，再試行をどこで扱うかに絞られる．

## 現時点で強い見立て

観察方法と取得方法は，別 plugin に分けず，同じ plugin が扱う前提で考えるのがよさそうである．

ただし，その情報を `channel` の直下属性として持たせるよりは，`channel` 本体の管理情報と収集実行設定を分けた方が見通しがよい．

現時点では，`channel` と `collector` のような分け方を仮置きするのがよさそうである．

- `channel`
  - 何を継続的に追うかを表す
  - 名前，状態，メモのような管理情報を持つ
- `collector`
  - その `channel` をどう観察し，どう取得するかを表す
  - 使用する plugin と，その plugin に渡す設定を持つ

この前提に立つと，`collector` 側にまず必要なのは次の 2 点で足りる可能性が高い．

- 使用する plugin の名称または識別子
- その plugin に渡す引数

ただし，引数を単に 1 つの塊として持つだけでは，

- 観察用の引数
- 取得用の引数

を区別しづらく，結果として `channel.crawlJobArgs` や `channel.recordingJobArgs` のような専用フィールドを増やしたくなりやすい．

そのため，引数の実体は plugin ごとに必要な形が異なりうるとしても，backend 本体の model としては，`jsonb` のような柔軟な構造の中に，少なくとも 1 階層の役割分けを持てるようにするのが自然である．

たとえば概念上は，

- plugin 識別子
- plugin 設定
  - 観察用設定
  - 取得用設定
  - 必要なら共通設定

のような形を想定するのがよい．

この整理を採ると，

- `channel` は「どの plugin で処理するか」と「その plugin に何を渡すか」を持つ
- plugin 側が，その引数から観察と取得の両方を実行する

という責務分担になる．

つまり，`endpoint` のような別 model を立てる代わりに，まずは `channel` に `collector` のような 1 階層をぶら下げ，その中で plugin と役割別設定を持てるようにする方向である．

## この整理の利点

- `channel` 本体を管理対象として素直に保てる
- 観察方法と取得方法の組を plugin 側へ閉じ込めやすい
- podcast のような単純なものでも，録画系のような複雑なものでも，同じ枠で扱いやすい
- `endpoint` のような中間概念を導入しなくても，まずは実装を進めやすい
- 役割別設定を `collector` 側へ押し込めることで，`channel` 直下の専用フィールド乱立を避けやすい

## この整理で残る注意点

- plugin 引数の妥当性を backend 側でどこまで検証するか
- `channel` 一つに対して `collector` が一つで足りる前提をどこまで置くか
- 同じ plugin の中で，観察用設定と取得用設定の語彙や必須性をどう定義するか
- `collector` の更新時に，plugin 引数の schema 変更へどう追随するか
- job 側が plugin を直接知るのか，常に `channel` を経由して引くのか

## ここまでの整理

現時点のメモから見ると，backend 側の情報収集は次の流れで捉えようとしている．

1. `channel` が継続観察の単位になる
2. `channel` にぶら下がる `collector` が観察方法でエントリー一覧を観測する
3. 新規の個別対象を予約エントリーとして保存する
4. 個別取得ジョブが実データを取得して backend に書き戻す
5. その結果を geshi 内で利用可能にする

また，この流れのうち，

- 観察方法
- 個別取得方法

は geshi 本体の固定実装ではなく，plugin に切り出したいという意図がある．

## まだ残っている主要論点

- `job` model から `target` を外すか，あるいは収集系では使わない前提にするか

# ADR-0009: データモデル仕様の正本を専用文書で管理する

## ステータス

決定

## 範囲

全体

## コンテキスト

- Geshi の次の主要な設計判断として，データモデルの整理が最重要になっている
- データモデルは，`backend`，`frontend`，`cli`，検索，収集，文字起こし結果の保存に広く影響する
- 一方で，データモデル仕様は実装や運用に合わせて段階的に具体化していく必要があり，個々の属性や関係の調整が比較的発生しやすい
- `決定` になった ADR の本文は原則として大きく書き換えない運用であるため，データモデルの本文そのものを ADR に閉じ込めると更新しづらい
- そのため，データモデルの内容そのものと，「どこを正本とするか」という設計判断を分けたい

## 決定

- データモデル仕様の正本は [`docs/models.md`](../models.md) とする
- Geshi のデータモデルに関する通常の更新は，`docs/models.md` に対して行う
- データモデルの大きな前提や位置づけを変更する場合は，新しい ADR で扱う

## 影響

- データモデルの通常更新を，`決定` 済み ADR を直接編集せずに行いやすくなる
- ADR は「データモデル仕様を専用文書で管理する」という判断を保持し，仕様本文は別ファイルで維持できる
- `docs/models.md` を参照すれば，現行のデータモデル仕様を 1 か所で確認できる

## 代替案

- データモデルを ADR 本文だけで管理する
  - 履歴は残しやすいが，通常更新がしづらい
- データモデルを実装コードだけで表現する
  - 実装前の設計判断や，複数層にまたがる共通理解を持ちづらい

## 備考

- `docs/models.md` の初期内容は，正本であることと想定対象を示す最小限の枠組みに留める
- 個別モデルの定義や関係は，後続の検討で `docs/models.md` に追記していく

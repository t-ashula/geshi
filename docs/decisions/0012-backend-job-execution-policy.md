# ADR-0012: backend のジョブ実行方針を定める

## ステータス

決定

## 範囲

`backend/`

## コンテキスト

- ADR-0004 により，`backend/` は収集，検索更新，外部 API 連携などのサーバ側責務を持つ
- ADR-0010 により，backend の HTTP API は Geshi における操作の共通入口として扱うことが決まった
- ADR-0011 により，永続化は結果整合性と冪等性を基本に設計する方針になった
- Geshi では，crawl，record，transcription 依頼など，API リクエストの同期処理だけでは扱いづらい長時間処理が想定される
- この段階で，ジョブ実行の位置づけと基本方針を定めておきたい

## 決定

- backend は，長時間処理や再試行が必要な処理をジョブとして扱えるようにする
- ジョブ実行は，HTTP API の同期処理とは分けて設計することを基本方針とする
- HTTP API は，ジョブの起動や参照の入口になれるようにする
- 外部スケジューラは，backend の API や CLI を起動するトリガに留める
- 同時実行数制御，重複起動防止，再試行，状態管理などの実行制御は backend に閉じる
- ジョブ実行は，結果整合性と冪等性を前提に設計する
- 同じ対象に対する同種ジョブの重複投入は，backend 側で制御できるようにする
- ジョブ状態は，少なくとも `queued`，`running`，`succeeded`，`failed`，`cancelled` のような共通語彙で扱えるようにする
- ジョブ自体は実行履歴と状態を表し，成果物そのものは domain データ側へ反映することを基本とする
- 少なくとも次の処理は，ジョブ化の対象として扱えるようにする
  - crawl
  - record / download
  - transcription 依頼
  - 検索インデックス更新
- queue，scheduler，worker の具体的な実装方式や採用ライブラリは本 ADR では決めない

### `record` の扱い

- この ADR でいう `record` には，リアルタイム録画や録音だけでなく，既存公開物の download も含める
- つまり，media 実体の取得と保存を伴う処理を，まとめてジョブ化対象として扱う

### ジョブ状態と結果の扱い

- ジョブは，domain データそのものではなく，実行履歴と制御対象として扱う
- たとえば `record` ジョブの結果として得られた `asset` や，`transcription` ジョブの結果として得られた `derivative` は，ジョブとは別に domain データとして保存する
- ジョブレコードには，状態，対象，起動時刻，完了時刻，失敗理由など，実行制御に必要な情報を持たせることを基本とする

## 影響

- API の応答時間と長時間処理を分離しやすくなる
- 再試行や再実行を前提にした設計へつなげやすくなる
- 後続の queue / scheduler 選定を，責務整理と分けて進められる

## 代替案

- 長時間処理もすべて HTTP API の同期処理で扱う
  - 実行時間や失敗時の扱いが厳しくなる
- 外部スケジューラから個別ジョブを直接制御する
  - 実行制御や状態管理が backend の外に漏れやすい
- ジョブ実行を backend の責務として明示せず，各機能で個別に扱う
  - 実行モデルがばらけやすい

## 備考

- 本 ADR は，ジョブ実行の位置づけと責務を定める
- queue，scheduler，実行基盤の具体選定は後続 ADR で扱う

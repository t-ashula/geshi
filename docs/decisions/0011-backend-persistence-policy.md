# ADR-0011: backend の永続化方針を定める

## ステータス

決定

## 範囲

`backend/`

## コンテキスト

- ADR-0004 により，`backend/` は保存対象メタデータの管理を責務として持つ
- ADR-0010 により，backend の HTTP API は Geshi における操作の共通入口として扱うことが決まった
- `docs/models.md` では，概念モデルと永続化実装を分けて考えることを明記している
- この段階で，backend が永続化をどう扱うかの基本方針を定めておきたい

## 決定

- backend は，概念モデルと永続化実装を分けて扱う
- 永続化実装は，概念モデルをそのまま 1 モデル 1 保存構造へ写すことを要求しない
- データの永続化実装では，特定ストレージ固有の高機能に過度に依存しない
- アプリケーション側で表現できる概念は，できるだけアプリケーション側のデータモデルやキー設計で表す
- 永続化と更新の設計では，できるだけ履歴を積み上げる形を取り，データをイミュータブルに保つことを基本とする
- 特に，基本データモデルと外部から取得したデータは，イミュータブルに保つことを原則とする
- 一方で，最新ステータスや現在値だけを持てばよい補助状態は，この原則の対象外とする
- 原則と例外の境界が曖昧なものは，後続の設計で都度判断する
- backend は，少なくとも次の永続化対象を扱えるようにする
  - `source` / `channel` / `feed` / `program` / `episode`
  - `asset` / `derivative`
  - 収集や処理の状態を表すメタデータ
- 永続化方式は，後続の詳細設計で必要に応じて複数の保存先を併用してよい
  - 例: 主 DB，検索インデックス，ファイルストレージ
- backend の永続化境界は，HTTP API やジョブ実行から見て一貫した読み書き単位になるように設計する
- DB の種類，migration ツール，ORM / query builder，検索インデックス方式の詳細は本 ADR では決めない

### 保存先固有機能への依存について

- 永続化の意味論を，特定のストレージ製品の専用機能だけに委ねない
- たとえばファイルのバージョニングが必要な場合，S3 のバージョニング機能にだけ依存するのではなく，バージョンに対応するキーやメタデータをアプリケーション側で扱える形を優先する
- これにより，保存先の変更やテスト環境での代替実装を行いやすくする

### 複数ストレージと整合性について

- 複数ストレージをまたぐ分散トランザクションは前提にしない
- 複数の保存先にまたがる更新では，結果整合性を目指す
- 更新処理や再処理は，適切に冪等性を担保できるように設計する
- 主データ，派生データ，検索インデックスの更新は，同時成功を前提にせず，再実行可能な形を優先する
- 永続化では，真実源として扱うデータと，そこから再構築可能な派生先を分けて考える
- どの保存先を真実源とし，どれを派生先とするかの具体的な対応は，後続の設計で決める

## 影響

- 概念モデルを保ったまま，保存構造は現実的に最適化できる
- 後続の DB 選定や検索基盤選定を，概念モデルの議論と分けて進められる
- API とジョブ実行の両方から見て，一貫した永続化境界を意識しやすくなる
- ストレージ製品の変更や代替実装に対する柔軟性を保ちやすくなる
- 履歴ベースかつ再実行可能な更新モデルを採りやすくなる

## 代替案

- 概念モデルをそのまま 1 モデル 1 テーブルで固定する
  - 初期理解はしやすいが，実装の自由度が低い
- 永続化方針をライブラリ選定と同時に決める
  - 技術選定に引きずられやすい

## 備考

- 本 ADR は永続化の責務と設計方針を定める
- 実際の DB 種別や migration 方針は後続 ADR で扱う

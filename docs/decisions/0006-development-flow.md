# ADR-0006: 開発項目駆動の開発フローを採用する

## ステータス

決定

## 範囲

全体

## コンテキスト

- Geshi は仕切り直し直後であり，実装より先に設計判断と終了条件を揃える運用が必要である
- ADR と Design log の運用方針はすでに存在するが，それをどの順序で使って開発を進めるかはまだ明文化されていない
- 文書と実装が混線すると，合意前にコードだけが先行しやすい
- 既存 ADR はすでに決定済みの設計判断として存在しており，新しい設計や実装はそれと無関係には進められない
- 一方で，あらゆる小さな作業に ADR を伴わせるのは過剰である
- 今回は，ADR を含みうる上位概念としての「開発項目」ごとに，設計判断，終了条件，実装，検証を明確に区切る流れを定めたい

## 決定

- Geshi の開発は，開発項目駆動のフローで進める
- 新しい設計や実装は，基本的に既存 ADR と矛盾しないように進める
- 既存 ADR を明確に更新したり，実質的に否定したりする場合は，影響範囲と理由を慎重に整理した上で，新しい ADR として扱う
- 基本の流れは以下とする

### 0. 開発項目を決める

- まず着手対象の開発項目を定める
- 開発項目は，必要に応じて ADR や Design log を伴いうる単位として扱う
- 細かな保守作業や軽微な変更まで，必ず ADR を伴わせる必要はない

### 1. ブランチを切る

- 開発項目ごとに作業ブランチを切る
- 複数の開発項目を 1 ブランチに混在させない

### 2. 必要な変更を ADR / Design log に起票する

- 重要な設計判断を伴う変更は ADR として起票する
- 必要に応じて 1 つの開発項目の中で複数の ADR に分ける
- 詳細な検討過程や比較，補足案は Design log に残す
- ADR にするほどではないが，判断の補足や検討過程を残したい場合も Design log を使ってよい
  - ADR とひもづかない Design log には番号として xxxx を固定する

### 3. 終了条件を文書に残す

- 必要に応じて，その開発項目の終了条件，受け入れ条件，検収要件を文書として残す
- 終了条件が曖昧なまま実装を始めない
- 終了条件を文書化する場合は `docs/acceptance/` を置き，対応する ADR と同名のファイル名で管理する
- 例:
  - `docs/decisions/0007-example.md`
  - `docs/acceptance/0007-example.md`

### 4. 文書だけでコミットする

- ADR，Design log，終了条件文書を作成した場合は，実装前に文書だけでコミットする
- まず「何を作るか」と「何をもって終わりとするか」を履歴に固定する
- これは文書とコードのコミットを極力混ぜないことで，実装をなかったことにしてやり直しやすくするためでもある

### 5. 実装を開始する

- コードや機能変更を伴う開発項目では，終了条件に基づいて実装を進める
- テスト先行を採る場合は，実装の中で TDD 的なサイクルとして回す
- 必要に応じてレビューや議論を行い，その内容は検収要件にも反映する
- その後にコードを実装する

### 6. 検収要件を満たしたら PR を作る

- 検収要件を満たすことを確認する
- `lint`，`format`，`test` などのチェックを通す
- PR を作成し，CI による検証を通す
- 問題がなければマージする

### 7. 次のタスクへ進む

- マージ後に次の開発項目を決め，同じフローを繰り返す

## 影響

- 実装前に設計判断と終了条件が固定される
- 文書とコードの責務が分かれ，レビューしやすくなる
- 開発項目単位で履歴を追いやすくなる
- CI による最終確認を開発フローに組み込みやすくなる
- 既存 ADR を軽く破って進めることを避けやすくなる
- ADR を伴う変更と，伴わない軽微な変更を分けて扱いやすくなる

## 代替案

- ADR を使わず，コードレビューの中で都度判断する
  - 初速は出るが，判断履歴が残りにくい
- 文書と実装を同じコミットで進める
  - 何を決めてから実装したのかが追いにくい
- 終了条件を文書化せず，PR 上の会話だけで済ませる
  - 後から受け入れ条件を追いにくい

## 備考

- 本 ADR は開発フローの基本手順を定めるものであり，ブランチ名規則や PR テンプレートの詳細までは定めない
- 終了条件の書式や Design log の書き方は，必要に応じて別途補う

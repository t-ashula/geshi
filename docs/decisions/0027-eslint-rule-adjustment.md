# ADR-0027: ESLint ルールの調整

## ステータス

決定

## 範囲

リポジトリ全体

## コンテキスト

- 現在は ESLint を導入済みで，最低限の lint は回っている
- 一方で，並び順や公開面，型定義の書き方など，実装中に毎回迷う点が残っている
- 開発を進める前に，どこまでを lint で強制し，どこからを人間の判断に残すかを整理したい
- コードが生成されることを前提として無意味な差分の発生とその認知負荷をさげたい

## 決定

- 現在の lint ルールを見直し，Geshi で強制したい規約だけを明示的に採用する
- 並び順や module member の sort を含む stylistic な規約も導入対象に含める
- backend / frontend / cli で共通化できる lint ルールは極力共通化する
  - frontend では Vue 固有の lint ルールや慣習を考慮する
  - 一般的な TypeScript / ESLint ルールと Vue の流儀が衝突する場合は，Vue 側を優先し，除外や上書きで調整する
- import の並び順，export の並び順，module 内の並び順，filename case，deep import 境界の制約も lint で扱う
- Geshi では formatter を導入してないので，format 関連の規約も ESLint 側で扱う
- 随時，ルールを見直すが方針としてはこの ADR を踏襲する
  - ルール変更を適用する際はコミットを分けるなどして機能のための変更とは区別する

## 影響

- 実装中に毎回迷う規約を減らせる
- 規約を人間の判断に委ねずに済む
- formatter を入れなくても書式を一定に保ちやすくなる
- frontend で Vue の書き方と lint が衝突しにくくなる
- import 境界や公開 API の使い方も lint で守りやすくなる

## 代替案

- 現状の ESLint 設定を据え置く
  - 実装ごとの迷いが残りやすい
- 最低限の安全性ルールだけに留める
  - 並び順や import 境界のような規約が人手依存のまま残る

## 備考

- 本 ADR は ESLint ルール調整の方針を対象とする
- 具体的な rule の追加・削除は後続の実装項目で扱う

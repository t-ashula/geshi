# ADR-0017: CLI ライブラリを選定する

## ステータス

決定

## 範囲

`cli/`

## コンテキスト

- ADR-0005 により，`cli/` は管理系の独立した入口であり，サブコマンド方式で機能を増やしていく方針になっている
- ADR-0010 により，backend の HTTP API は Geshi における操作の共通入口として扱う
- CLI は最終的には backend API を呼ぶ薄い入口に寄せたいが，その前提でもサブコマンド定義やヘルプ，オプション処理のためのライブラリは必要になる
- まずは過度に重くない形で CLI を初期化できるライブラリを選びたい

## 決定

- `cli/` のライブラリとして `commander` を採用する
- 採用理由は次の通りである
  - サブコマンド方式と相性がよい
  - TypeScript で扱いやすい
  - 導入が過度に重くない
  - コマンド構成やディレクトリ構成をフレームワークに強制されにくい
- `npm install` や初期実装は後続の開発項目で扱う

## 影響

- `cli/` の初期実装を進めやすくなる
- backend API 呼び出しとの対応づけを考えやすくなる
- コマンド体系の具体化を進めやすくなる
- 最初は `commands/*.ts` のような素直な構成で始めて，必要になった時点でコマンド単位に分割しやすくなる

## 代替案

- `yargs` を採用する
  - 機能は十分だが，現段階では `commander` の方が軽く始めやすい
- `cac` を採用する
  - 軽量だが，サブコマンドを増やしていく前提では `commander` の方が安心しやすい
- ライブラリ選定を後ろへ送る
  - `cli/` の初期実装が進みにくい

## 備考

- 本 ADR は `commander` の採用判断を記録する
- 候補の比較や細かい検討は，必要なら Design log ではなく `tmp/` に置いてもよい

# ADR-0002: 開発項目駆動の開発フローを採用する

## ステータス

決定

## 範囲

全体

## コンテキスト

- ADR と Design log の運用方針はすでに存在するが，それらを使って開発を進めには終了条件を明確にしておく必要がある
- 一方で，あらゆる小さな作業に ADR を伴わせるのは過剰である
- ADR を含みうる上位概念としての「開発項目」ごとに，設計判断，終了条件，実装，検証を明確に区切る流れを定めたい

## 決定

開発項目駆動のフローで進める

1. 作業用にブランチを作成しスイッチ
2. このブランチで終了条件（マージ受け入れ条件）を決める
3. 受け入れ条件達成に必要な ADR を起票の後，詳細を議論し，決定にするか却下する
4. 決定にしたADR，必要に応じてテストを書き，開発作業をする
5. 受け入れ条件を満たすまでテストや動作検証と開発作業を行う．
   - 必要に応じて追加の ADR を起票したり，細分化した受け入れ条件としてブランチを分岐したりして，作業を続ける
6. 条件が満たされたらベースブランチにマージする

数コミット程度で終わる，細かな保守作業や軽微な変更では，このフローに合わせなくて良い．
詳細は [development](../development.md) のフローのセクションに記す．

## 影響

- 実装前に設計判断と終了条件が固定される
- 文書とコードの責務が分かれ，レビューしやすくなる
- 開発項目単位で履歴を追いやすくなる
- CI による最終確認を開発フローに組み込みやすくなる
- 既存 ADR を軽く破って進めることを避けやすくなる
- ADR を伴う変更と，伴わない軽微な変更を分けて扱いやすくなる

## 代替案

- ADR を使わず，コードレビューの中で都度判断する
  - 初速は出るが，判断履歴が残りにくい
- 文書と実装を同じコミットで進める
  - 何を決めてから実装したのかが追いにくい
- 終了条件を文書化せず，PR 上の会話だけで済ませる
  - 後から受け入れ条件を追いにくい

## 備考

- 本 ADR は開発フローの基本手順を定めるものであり，ブランチ名規則や PR テンプレートの詳細までは定めない
- 終了条件の書式や Design log の書き方は，必要に応じて別途補う

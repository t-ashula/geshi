# ADR-0005: cli の初期アーキテクチャ方針

## ステータス

決定

## 範囲

`cli/`

## コンテキスト

- Geshi の `cli/` は管理コマンド，運用コマンド，保守コマンドの入口になる
- CLI は backend の内部機能を呼び出すことが多く，言語や型を揃えておきたい
- 一方で，CLI フレームワークやコマンド体系は，開発フローや運用方針を決めてから詰めたい

## 決定

- `cli/` は管理系の独立した入口として扱う
- 初期段階では，[adr-0002] にもとづいて以下の基盤だけを先に整える
  - Node.js
  - TypeScript
  - ESLint
- CLI の機能追加はサブコマンド方式で積み上げていく前提にする
- `cli/` は backend のユースケースを呼び出す側に留め，CLI 独自の業務ロジックは極力持たない
- 将来的には，CLI から backend の HTTP API を呼ぶだけで成立する形へ寄せる
- CLI フレームワークや引数パーサは本 ADR では決めない
- まずは backend と分離された CLI 実装の受け皿を用意することを優先する

### 現時点で決めないこと

- CLI フレームワークの採用
- 出力フォーマットや UX
- backend と共有するライブラリ境界

## 影響

- backend と CLI の責務を物理的に分離できる
- 実装言語を統一しつつ，CLI 設計の判断を後続へ送れる
- 管理系の入口を早期に確保できる
- 将来の機能追加を `geshi <domain> <action>` のような形で整理しやすくなる
- CLI が薄い制御層として保たれやすくなる

## 代替案

- CLI を backend に内包する
  - 管理系コードとサーバ系コードが混ざりやすい
- CLI を shell script 中心で運用する
  - 初速は速いが，保守性が落ちやすい

## 備考

- 本 ADR は CLI の責務範囲と初期基盤だけを定める
- CLI ライブラリの選定や詳細なコマンド設計は別 ADR で扱う

# ADR-0004: 開発環境の初期設定を標準化する

## ステータス

決定

## 範囲

全体

## コンテキスト

- `geshi` は `web ui frontend`，`api backend`，`crawler`，`cli`，`storage` を含む構成を前提としている
- これらを開発していくには，各開発者がローカルで揃える前提を最初に決めておく必要がある
- 初期設定が人ごとにばらつくと，再現性のない不具合や，セットアップの属人化が起きやすい
- 一方で，実装言語や個別ツールの細部は今後の決定に委ねたい

## 決定

開発環境の初期設定として，当面は以下を標準とする

### 1. ツールバージョンの固定

- 言語ランタイムや主要 CLI ツールのバージョンは `.tool-versions` で管理する
- `.tool-versions` は asdf 互換ツールで扱える形式を前提とする
- 個々の開発者は asdf または互換ツールを用いて必要バージョンを揃える

### 2. ローカル依存サービスの起動

- DB，検索エンジン，オブジェクトストレージなどのローカル依存サービスは `docker compose` で起動する
- アプリケーション本体の起動方法とは分けて管理する
- 開発者が最初に必要なサービス群を一括で起動できる状態にする

### 3. ローカル設定と秘密情報

- 開発者ごとの差異を持つ設定や秘密情報は `.env.local` のようなローカル専用ファイルで扱う
- リポジトリには `.env.example` のような雛形を置き，必要な設定項目だけを共有する
- 実値の秘密情報はコミットしない

### 4. 日常操作の入口

- セットアップ，起動，テスト，Lint などの日常操作はリポジトリルートから実行できる入口を用意する
- 具体的な入口は `make`，`just`，`npm run` など今後採用する方式に委ねる
- 少なくとも README または開発文書から，同じ手順に辿れるようにする

### 5. Git の初期設定

- コミットメッセージは既存 ADR に従い `.commit-template.txt` を利用する
- 各開発者は `git config commit.template .commit-template.txt` を設定する

詳細なセットアップ手順は [development] に記す．

## 影響

- 開発者ごとの差異を減らし，初回セットアップ手順を揃えやすくなる
- 依存サービスの再現性を保ちやすくなる
- 実装言語やフレームワークの決定前でも，最低限の開発環境方針を先に固定できる

## 代替案

- 各開発者が自由に環境を整える
  - 初速は出るが，再現性が落ちやすい
- すべてをコンテナ内開発前提にする
  - 一貫性は高いが，現段階では重い
- ランタイム管理方法を決めず，README の説明だけに留める
  - 手順は残るが，バージョンの揺れを防ぎにくい

## 備考

- 本 ADR は初期設定の標準を定めるものであり，具体的な言語やフレームワークまでは定めない

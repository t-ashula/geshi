# ADR-0030: データモデル

## ステータス

決定

## 範囲

全体

## コンテキスト

- [adr-0009], [adr-0010] やその後続の ADR で十分な議論や要件を示さずにデータモデルの名称と関係性だけが記録されていったために収集がつかなくなった
- 要件から導かれる概念的なモデルとその関係性といった概念の整理とそれぞれに必要な情報の設計， DB のテーブルなどの物理的な構造からくる設計との分離が不十分

## 決定

- モデル設計と文書化に層分離原則 [principal] を導入する
  - [adr-0009] で行った ADR と [models] の関係のより具体的な文書化でもある
  - 概念的なモデルとその関係については ADR で固定する
  - 必要な情報など具体的なインタフェースは [models] に記述する
  - テーブルやファイルでの永続化の物理的な構造はコードで示す
- これまでの [models] は一度白紙にする

## 影響

- `ADR-0030` 自体では，個別の model 名や関係を決めない
- 後続の model 整理では，概念の固定と具体的インタフェース記述と物理実装を混ぜずに進める必要がある

## 代替案

- `0030` の中で model 一覧や関係までまとめて決める
  - ただし範囲が広すぎ，概念整理と具体インタフェース設計と実装都合が混ざりやすいので採らない
